# PixelForge - Rust/WASM ピクセルアートエディタ 要件定義書

**バージョン:** 1.1
**作成日:** 2026年3月
**最終更新:** 2026年3月（レビュー指摘反映）
**ステータス:** Draft

---

## 1. プロジェクト概要

### 1.1 目的

Rust + WebAssembly によるブラウザ完結型のピクセルアートエディタを開発する。  
既存ツール（pixcil 等）の基本機能に加え、**AI による画像解析・提案機能**を統合することで、  
コンセプトイメージや参照画像からピクセルアートを効率よく制作できるワークフローを実現する。

### 1.2 対象ユーザー

- インディゲーム開発者（Bevy/Unity/Godot 向けスプライト制作）
- ピクセルアートクリエイター（SNS 発信・NFT 等）
- AI アシストを活用したい初中級者

### 1.3 技術スタック

| レイヤー | 技術 |
|---|---|
| コアロジック | Rust |
| ブラウザ実行 | WebAssembly (wasm-pack / wasm-bindgen) |
| UI レンダリング | WebGL 2.0（レイヤー合成・ブレンド）+ Canvas 2D API（UI オーバーレイ） |
| フロントエンド | Leptos（Rust フル WASM）— UI・状態管理を Rust で統一 |
| AI 連携 | Anthropic Claude API / OpenAI Vision API（外部呼び出し） |
| ビルド | trunk |
| ストレージ | IndexedDB（プロジェクト保存）、LocalStorage（設定） |

> **技術選定の経緯:**
> - **レンダリング:** Canvas 2D では複数レイヤー × ブレンドモード（乗算/スクリーン等）を 2048×2048 で 60fps 維持が困難なため、WebGL 2.0 を採用。WebGL 非対応環境では Canvas 2D へのフォールバックを用意する。
> - **フロントエンド:** Leptos を採用し Rust で統一することで、コアロジックとの型共有・シリアライズコスト削減を実現する。Phase 0（技術検証）で Leptos + WebGL の PoC を実施し、パフォーマンス要件を満たすことを確認する。

## 2. アーキテクチャ設計

### 2.1 モジュール構成

```
pixelforge/
├── crates/
│   ├── core/           # キャンバス・ピクセルデータ・描画ロジック
│   ├── tools/          # ブラシ・消しゴム・塗りつぶし等ツール実装
│   ├── formats/        # PNG/GIF/Aseprite インポート/エクスポート
│   ├── animation/      # フレーム・タイムライン管理
│   ├── renderer/       # WebGL 2.0 レンダリング（レイヤー合成・ブレンド）
│   └── wasm_bridge/    # wasm-bindgen エントリポイント、JS FFI
├── app/                # Leptos UI アプリケーション（コンポーネント・状態管理）
├── styles/             # CSS
└── ai_service/         # AI 連携サーバーレス関数
```

### 2.2 データフロー

```
ユーザー操作（Mouse/Touch/Keyboard）
        ↓
   Leptos イベントハンドラ（Rust）
        ↓
   wasm_bridge ← undo/redo スタック管理
        ↓
   core::Canvas（ピクセルバッファ）
        ↓
   renderer::WebGL（レイヤー合成・ブレンドモード処理）
        ↓
   <canvas> 要素（ブラウザ表示）

※ WebGL 非対応環境では Canvas 2D API へ自動フォールバック
```

## 3. 機能要件

### 3.1 基本描画ツール

| 機能 | 詳細 |
|---|---|
| ペンシル | 1px〜16px、アンチエイリアスなし |
| 消しゴム | サイズ可変 |
| 塗りつぶし | フラッドフィル、許容値設定可能 |
| スポイト | カラーピッカー兼用 |
| 矩形選択 | コピー・移動・削除 |
| 投げなわ選択 | 任意形状選択 |
| ライン / 矩形 / 楕円 | プリミティブ描画 |
| テキスト | ピクセルフォント入力 |
| グラデーション | ディザリングによるピクセルグラデーション |

### 3.2 キャンバス管理

- キャンバスサイズ: 8×8 〜 512×512（推奨）、最大 2048×2048
- メモリバジェット: キャンバスサイズ × レイヤー数 × 4bytes(RGBA) の合計が 256MB 以下
  例: 512×512 × 20 レイヤー ≈ 20MB、2048×2048 × 4 レイヤー ≈ 64MB
  上限超過時はレイヤー追加を警告付きで制限
- ズーム: 1x 〜 64x（スムーズズーム）
- グリッド表示：ピクセルグリッド / タイルグリッド切り替え
- パン: スペース + ドラッグ、またはミドルドラッグ
- ミラー描画（左右 / 上下対称）

### 3.3 レイヤー管理

- レイヤー追加 / 削除 / 複製 / 並び替え
- 不透明度スライダー（0〜100%）
- ブレンドモード: 通常 / 乗算 / スクリーン / オーバーレイ
- レイヤーグループ（フォルダ）
- レイヤー表示 / ロック切り替え
- 参照レイヤー（下絵として使用、AI インポート画像はここに配置）

### 3.4 カラー管理

- パレット: 最大 256 色
- HSV / RGB / HEX 入力
- パレットインポート: GIMP GPL / Adobe ACO / Lospec JSON
- パレットエクスポート
- カラー履歴（直近 32 色）
- 減色処理（画像インポート時の自動パレット生成）

### 3.5 アニメーション

- フレームベースアニメーション（GIF / スプライトシート兼用）
- タイムライン UI（フレーム追加 / 削除 / 複製 / 並び替え）
- FPS 設定（1〜60）
- オニオンスキン（前後フレームの半透明表示）
- ループ再生プレビュー

### 3.6 インポート / エクスポート

| 形式 | インポート | エクスポート |
|---|---|---|
| PNG | ✅ | ✅（透過対応） |
| GIF | ✅（アニメーション） | ✅ |
| JPEG | ✅（参照用） | ✅ |
| WebP | ✅ | ✅ |
| Aseprite (.ase/.aseprite) | ✅（Phase 3 以降） | ✅（Phase 3 以降） |
| スプライトシート | ✅（グリッド指定分割） | ✅（レイアウト設定） |
| SVG | — | ✅（低優先度・Phase 4 以降） |
| JSON（プロジェクト） | ✅ | ✅ |

> **優先度の考え方:**
> MVP（Phase 1〜2）では PNG / GIF / JSON に集中する。Aseprite は仕様が複雑（圧縮・ブレンドモード）なため Phase 3 以降で `asefile` クレートを活用して実装する。SVG エクスポートはユースケースが限定的なため低優先度とする。

### 3.7 Undo / Redo

- Undo 上限: メモリ使用量 128MB または 操作 500 回のいずれか先に到達した時点で古い履歴から破棄
- コマンドパターンによる差分保存（全ピクセルコピーではなく変更差分のみ保持）
- Undo 履歴パネル（操作名一覧）
- メモリ上限に近づいた際のユーザー通知

## 4. AI 機能要件（コア差別化機能）

### 4.1 画像インポート → AI 解析・提案パイプライン

```
① ユーザーが参照画像をドロップ / ファイル選択
        ↓
② フロントエンドが画像を Base64 エンコードして API 送信
        ↓
③ AI (Claude Vision / GPT-4V) が画像解析
        ↓
④ 以下のいずれかを返却（ユーザーが選択）
   ・ A) ピクセルアート変換（自動減色・ドット絵化）
   ・ B) カラーパレット提案
   ・ C) スタイル提案（テキスト + 見本）
   ・ D) 構図・リファレンス分析コメント
        ↓
⑤ 結果をキャンバスに適用（参照レイヤー or 新規レイヤー）
```

### 4.2 自動ドット絵変換 (AI-Assisted Pixelization)

**入力:** 任意の写真 / イラスト / コンセプトアート
**出力:** 指定解像度（16×16 〜 128×128）のピクセルアート

> **AI とローカル処理の役割分担:**
> - **AI の役割（テキストベース）:** スタイル分析・パラメータ提案・修正アドバイスの返却。AI はピクセルデータを直接生成しない。
> - **Rust の役割（ローカル処理）:** AI の提案パラメータに基づき、実際の減色・ダウンサンプリング・ディザリングを実行する。
> - この分担により、処理の再現性とレスポンス速度を確保する。

処理フロー:

1. 解像度指定（ターゲットサイズ選択）
2. パレット制約指定（カラー数上限: 4 / 8 / 16 / 32 / 64）
3. AI が画像を解析し、以下をJSON形式で返却:
   - 推奨スタイル（RPG ゲーム風 / アーケード風 / ミニマル 等）
   - 推奨ディザリング手法（None / Floyd-Steinberg / Ordered 等）
   - 推奨パレット（HEX 値の配列）
   - アウトライン処理の推奨（太さ・色）
4. Rust 側で AI の提案パラメータを元に減色 + ダウンサンプリング処理を実行（Nearest Neighbor / 知覚的量子化）
5. AI が追加アドバイス（「アウトラインを強調すると良い」等）をサイドパネルに表示
6. ユーザーはパラメータを手動調整して再処理可能（AI 再呼び出し不要）

### 4.3 カラーパレット自動提案

- インポート画像から主要な色を抽出し、ピクセルアート向けに最適化したパレットを提案
- 提案パレット候補を複数表示（例: 「夕焼け系」「モノクロ+アクセント」等）
- ワンクリックで現在のパレットに適用

### 4.4 スタイル提案・テキスト指示

- テキストプロンプトで「ゲームボーイ風」「16-bit JRPG 風」等のスタイルを指定
- AI が現在のキャンバス画像を解析し、スタイルに合わせた修正提案をコメントで返す
- 提案内容: 色調・コントラスト・アウトライン・ディザリング技法 等

### 4.5 スプライト補完・生成提案

- 既存スプライトの歩行アニメーションの未作成フレームを AI が補完提案
- キャラクタースプライトの左向き → 右向きのミラー + 微調整を AI が提案
- タイルセットのパターン補完提案

### 4.6 AI 機能の UI 設計

```
┌─────────────────────────────────┐
│  AI アシスタントパネル          │
├─────────────────────────────────┤
│  [画像をドロップして解析]        │
│                                 │
│  解析モード:                    │
│  ○ 自動ドット絵変換             │
│  ○ パレット提案                 │
│  ○ スタイル分析                 │
│  ○ テキスト指示                 │
│                                 │
```

### 4.7 AI API 設計

- **バックエンド:** サーバーレス関数（Cloudflare Workers / AWS Lambda）
  ※ API キーをクライアントに露出させないため
- **エンドポイント:**
  - `POST /api/analyze` — 画像解析・パレット提案
  - `POST /api/pixelate` — AI アシスト変換指示取得
  - `POST /api/suggest` — テキストプロンプト → スタイル提案
- **認証:** 必須。匿名トークン（デバイスフィンガープリント + IP）による最低限の認証を実装。将来的にソーシャルログイン（GitHub / Google）の追加を検討
- **レート制限:** トークン単位で管理（1 時間あたり 20 リクエスト、1 日あたり 100 リクエスト）
- **コスト管理:**
  - Vision API は 1 リクエストあたり入力トークンが大きいため、画像をリサイズ（最大 1024px）してから送信
  - ユーザーごとの月間クォータ上限を設定（無料枠: 50 回/月）
  - API コストダッシュボード（管理者向け）
- **プライバシー:** 画像は処理後即時破棄、ストレージ不使用

### 4.8 AI 機能のオフライン時の挙動

- AI パネルはオフライン時も表示するが、操作ボタンを無効化しオフラインである旨のメッセージを表示
- ローカル処理のみで動作する機能（パラメータ手動調整による再処理）はオフラインでも利用可能
- オンライン復帰時に自動的に操作を再有効化

## 5. 非機能要件

### 5.1 パフォーマンス

| 指標 | 目標値 |
|---|---|
| 初回ロード時間 | < 2 秒（WASM バンドル含む） |
| 描画レスポンス | < 16ms（60fps 維持、WebGL 使用時） |
| キャンバス最大サイズ | 2048×2048 px |
| レイヤー上限目安 | 512×512: 20 枚、2048×2048: 4 枚（メモリバジェット 256MB 以内） |
| Undo 処理時間 | < 50ms |
| Undo メモリ上限 | 128MB または 500 操作 |
| AI 解析レスポンス | < 10 秒（目標 3〜5 秒） |

### 5.2 対応環境

- **ブラウザ:** Chrome 90+ / Firefox 88+ / Safari 15+ / Edge 90+
- **デバイス:** PC（主）、タブレット（タッチ操作対応）
- **オフライン動作:** AI 機能以外はフル動作（PWA 対応）。AI パネルはオフライン時に無効化表示

### 5.3 セキュリティ

- AI API キーはサーバーサイドのみで保持
- ユーザー画像データの外部送信は明示的な同意後のみ（初回利用時に同意ダイアログ表示）
- CORS 適切設定
- Content Security Policy 設定
- AI API 認証: 匿名トークン + IP 制限によるアクセス制御

### 5.4 ストレージ制限

- IndexedDB の使用上限: プロジェクトあたり最大 50MB、全体で 500MB を推奨上限とする
- 上限に近づいた際にユーザーへ警告を表示し、エクスポート・削除を促す
- 自動保存: 操作 30 秒ごと、またはフォーカス喪失時にクラッシュリカバリ用の自動保存を実行

### 5.5 アクセシビリティ

- すべてのパネル・ボタンにキーボードナビゲーション対応（Tab / Enter / Escape）
- ツールバー・パネルに適切な ARIA ラベルを付与
- 高コントラストテーマの提供（OS 設定連動）
- カラーピッカーでの色情報テキスト表示（HEX / RGB 値の読み上げ対応）

### 5.6 国際化 (i18n)

- 初期リリース: 日本語 / 英語の 2 言語対応
- UI テキストは言語リソースファイル（JSON）で外部管理し、追加言語に対応可能な設計とする
- ブラウザの `navigator.language` に基づく自動言語選択 + 手動切り替え

### 5.7 エラーハンドリング

- WASM パニック時: `console_error_panic_hook` でエラーログを出力し、ユーザーにリカバリダイアログを表示（自動保存データからの復元を提案）
- ネットワークエラー（AI API 呼び出し失敗時）: リトライボタン付きエラーメッセージをパネル内に表示。ローカル処理には影響しない
- ファイル読み込みエラー: 対応形式・サイズ制限を明示したエラーメッセージを表示

### Phase 2 — エディタ完成（約 6 週間）

- 全ツール実装
- カラーパレット管理
- アニメーション / タイムライン
- GIF エクスポート
- キーボードショートカット
- タイル / スプライトシートサポート

### Phase 3 — AI 機能 + 追加フォーマット（約 8 週間）

- PWA 化（オフライン対応・Service Worker）
- パフォーマンスチューニング
- IndexedDB プロジェクト保存（容量管理・警告表示）
- SVG エクスポート（低優先度、余裕があれば）
- ユーザーテスト・フィードバック反映

## 8. 技術的な注意点・リスク

### 8.1 WASM バンドルサイズ

- Rust の画像処理クレート（image crate 等）を使うとバンドルが肥大化しやすい
- 対策: `wasm-opt` による最適化、feature フラグで不要コーデック除外、コードスプリット

### 8.2 レンダリングパフォーマンス

- WebGL 2.0 で各レイヤーをテクスチャとして管理し、ブレンドモードはシェーダーで処理
- 大きいキャンバス（512px 以上）ではダーティリージョン管理で再描画範囲を最小化
- Canvas 2D フォールバック時はオフスクリーン Canvas + `transferToImageBitmap` を活用
- `SharedArrayBuffer` + `Worker` でピクセル処理をバックグラウンド実行を検討

### 8.3 AI API レイテンシ

- Vision 系 API の応答は 3〜10 秒かかる場合がある
- ストリーミングレスポンスで段階的に結果表示し、UX を改善

### 8.4 Aseprite 互換性

- Aseprite フォーマットの仕様はオープンだが複雑（圧縮・ブレンドモード）
- `asefile` クレートの活用を推奨
- Phase 3 以降で対応し、MVP では PNG/GIF に集中

### 8.5 WebGL 非対応環境

- WebGL 2.0 の利用率は対象ブラウザで 97% 以上だが、一部の古い環境やハードウェアアクセラレーション無効時にフォールバックが必要
- Canvas 2D フォールバック時はブレンドモード（乗算/スクリーン等）を手動で実装する必要がある（パフォーマンス低下を許容）

## 9. テスト戦略

### 9.1 単体テスト（Rust）

- core / tools / formats / animation クレートに対して `#[cfg(test)]` によるユニットテストを実装
- ピクセルバッファ操作、色変換、Undo/Redo ロジック等の純粋なロジックを重点的にカバー
- カバレッジ目標: コアロジック 80% 以上

### 9.2 WASM 統合テスト

- `wasm-bindgen-test` を使用し、ブラウザ環境での WASM 呼び出しをテスト
- WebGL / Canvas 2D の描画結果はスナップショットテスト（ピクセル単位比較）で検証

### 9.3 E2E テスト

- Playwright を使用し、主要ユーザーフロー（描画・保存・読み込み・Undo/Redo）を自動テスト
- AI 機能はモックサーバーを使用してテスト

### 9.4 パフォーマンステスト

- Phase 0 の PoC で確立したベンチマーク（512×512 / 10 レイヤー / 60fps）を CI で継続監視
- WASM バンドルサイズの上限チェック（目標: gzip 後 1MB 以内）
